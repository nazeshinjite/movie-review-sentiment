# LLM-as-Judge — Development Log

Running record of process, measurements, and decisions for notebook `05_divergence_judge` (Ian, T3). Append-only, newest date last. Each entry: what was done → what was observed → what was decided. Source material for the paper's Methods section, the AI-use disclosure, and presentation talking points.

**Fixed setup for all entries unless noted:** judge model Qwen3.6-35B-A3B (Q8_0 MoE, ~3B active params, native MTP speculative decoding) served locally via llama.cpp (Homebrew build 9960) at an OpenAI-compatible endpoint; `temperature=0` on every request; endpoint configured via the repo's gitignored `.env`.

## 2026-07-20

- **Authored the judge development plan** around a test-against-the-contract strategy: because all prediction files share one schema (`id, y_true, y_pred, y_proba_pos`), the judge pipeline can be built and fully tested against stand-ins — (1) a hand-labeled golden set for prompt quality, (2) stub predictions from throwaway classifiers for plumbing, (3) a full stub dry-run for scale — then swap to the real model outputs with only a file-path change.
- **Decided: golden set sourced from the IMDB *train* split** (not val, deviating from the original plan wording) so prompt tuning never touches data that later feeds the real adjudication run.
- **Decided: blind adjudication** — the judge never sees which label LR or the NN predicted, so its verdict is an independent third opinion with no anchoring.
- **Configured the endpoint** (`.env`: base URL, key, model alias; key never enters the public repo) and **smoke-tested the API round trip**: one completion through the OpenAI client returned clean in **1.57 s**; the server correctly separates Qwen's reasoning stream (`reasoning_content`) from the answer (`content`).
- **Verified schema enforcement (harness requirement)**: `response_format` with a JSON-schema forced-binary enum (`{"label": "positive"|"negative"}`) returned valid, parseable JSON in **all four tested conditions** — short/long review × reasoning on/off. The server applies the grammar to the answer *after* free-form reasoning, so structured output and reasoning mode compose. Decision: the judge's output is schema-constrained forced-choice JSON — malformed-output handling is unnecessary by construction.
- **Verified the reasoning toggle**: `chat_template_kwargs: {"enable_thinking": false}` cleanly disables reasoning (~2,900 chars of reasoning when on → 0 when off), with schema-valid output in both modes.
- **Measured per-call latency** (2×2: ~30-word vs ~424-word review × reasoning on/off, temp 0):

  | Condition | Latency | Completion tokens |
  |---|---|---|
  | reasoning ON, short | 7.18 s | 782 |
  | reasoning ON, long | 6.74 s | 778 |
  | reasoning OFF, short | 0.22 s | 12 |
  | reasoning OFF, long | 0.40 s | 12 |

  Observations: reasoning costs ~**20×** per call (~770 reasoning tokens regardless of review length); prompt length is nearly free (+0.18 s per ~490 prompt tokens). Projected onto the expected ~1,000–2,000 test-set disagreements: **~10–20 min with reasoning off vs ~2–4 h with reasoning on**.
- **Decision rule set for reasoning mode:** it will be A/B-tested on the golden set and must earn its ~20× latency cost with measurably better hard-case accuracy (> ~2 points), otherwise it stays off. All four sanity labels in the latency test were correct.
- **Added `openai` to the repo environment** (`uv add openai` → pyproject + lockfile change riding the `ian/05-divergence-judge` branch, so the dependency lands in the same PR as the notebook that needs it, per the deferred-deps policy).
- **Built the golden set (B1): 31 IMDB train-split reviews, hand-curated** from 48 pattern-mined candidates (negation phrasings; mixed-signal hedges; sarcasm mined as negative-labeled reviews containing praise words). Composition: 17 negative / 14 positive; 9 easy (clear polarity) / 22 hard (8 negation, 6 mixed, 5 sarcasm, 3 short/noisy); median 296 words. Two candidates were re-categorized on human read ([17596] gold-positive but lukewarm → mixed; [18856] praise-via-negations → negation). Train-split sourcing means prompt tuning never touches evaluation data. Saved as `notebooks/05/golden_set.csv`.
- *Environment note:* a stale Hugging Face token caused 401s on a public dataset; runs use `HF_HUB_DISABLE_IMPLICIT_TOKEN=1` until the machine's token is refreshed.
- **Golden set human-reviewed (Ian) and corrected:** [23913] recategorized mixed → clear_pos (reads as a straightforward positive); [2548] removed (ambiguous between clear_neg and sarcasm — category-ambiguous items muddy a stratified readout) and replaced with [10617] (unambiguous pan); [8500] removed (plain disappointment, not irony) and replaced with [1361] (sustained derision). Frozen composition: 31 reviews, 17 negative / 14 positive, 10 easy / 21 hard (8 negation, 5 mixed, 5 sarcasm, 3 short/noisy), median 308 words. Gold labels themselves are untouched throughout — they are the dataset's (the reviewers' own star ratings), and redefining them would redefine the task.
- **Prompt v1 evaluated on the golden set (A/B: reasoning on vs off)**, temp 0, schema-constrained, blind. Verbatim system prompt:

  > *"You are a sentiment adjudicator for movie reviews. Read the review and classify the reviewer's overall opinion of the film as positive or negative. You must choose exactly one, even for mixed reviews. Respond in JSON."*

  Results (final reviewed set, frozen config): **30/31 = 96.8% overall, 20/21 = 95.2% on the hard subset** — clears the pre-set bar (≥90% / ≥80%) with no prompt iteration, so no few-shot variants were pursued (iterating on a 96.8% over 31 items would be overfitting the instrument). Reasoning on vs off produced **identical predictions** (same 30 correct, same 1 miss) at ~16× the wall-clock (208 s vs 12–13 s for 31 reviews).
- **Decision: reasoning OFF and prompt v1 FROZEN** (zero-shot, blind, forced-binary, temp 0). Per the pre-registered rule, reasoning had to beat by >2 points on hard cases to justify ~20× latency; it tied exactly. Projected test-day run (~1,000–2,000 disagreements): roughly 7–15 minutes.
- **The single miss is informative, not fixable:** [17596], the label-noise exemplar — prose reads lukewarm-negative ("Hardly a masterpiece… B- and that's being generous") but the author's own rating makes it gold-positive. The judge read the text reasonably; the gold reflects the rating, not the prose. Kept in the set deliberately as the irreducible-noise case; worth citing in the paper's discussion.
- **Tagging protocol drafted (D1, v0.1)** at `notebooks/05/tagging-protocol.md`: five categories (negation / sarcasm / mixed / noise / other) with decision rules, anchors from the golden set, precedence order, blind-tagging procedure, and the pre-registered sampling rule. Distinction clarified after review discussion: the protocol's `noise` (defective *text* — no arguable verdict) is not "label noise" (readable text whose *gold label* surprises, e.g., [17596], which tags `mixed`).
- **Mixed-vs-noise boundary sharpened and the golden set expanded to 32** with one true noise probe, [21360] (abstract art-prose with no arguable verdict). Heuristic mining for noise (opinion-free synopses, non-English, markup-heavy) yielded a pre-registerable empirical finding: **genuine noise is rare in this corpus after markup stripping** — two of three candidate specimens failed the protocol's own assessability test on full read ([9652]'s verdict is merely buried under ~550 words of synopsis → cited in the protocol as the `other` illustration; [745] is plainly negative despite a truncated opening). Expect a low `noise` count in the real disagreement set.
- **Re-score on the final 32-review instrument (frozen config): 31/32 = 96.9% overall, 21/22 = 95.5% hard.** The judge called the noise probe correctly. Same single miss ([17596]).
- **Tagging protocol ratified to v1.0 (Ian):** one primary tag per review; precedence `noise > sarcasm > negation > mixed > other`; blind tagging; sampling rule all-if-≤50 else seeded random 50. Added an Inputs section specifying how the blinded tagging sheet is derived (merge predictions on `id`, filter `y_pred` mismatch, join text from splits, strip gold/votes/verdict before tagging) and the honest pilot caveat (pilot is semi-blind since the tagger curated the golden set; the real test-split run is fully blind). D1 complete; D2 pilot remains.

## 2026-07-22

- **Repo layout standardized (pre-commit):** `notebooks/` holds only the six pipeline notebooks, per the handoff contract. The judge lane's supporting materials moved out of the interim `notebooks/05/` folder: this dev log → `docs/judge-dev-log.md`, the tagging protocol → `docs/tagging-protocol.md`, and the two hand-curated CSVs (golden set, blinded pilot sheet) → `data/golden/` — committed, unlike `data/raw|processed`, because hand-curated files cannot be regenerated. Earlier entries reference the old `notebooks/05/` paths; they are left as written.
- **D2 pilot tagging run (Ian):** all 32 golden reviews tagged blind via a small terminal tool (one review at a time, choices presented in precedence order, per-review timing captured). 16 min total, median 19 s/review; 3 hesitations recorded as `tag_alt`, all three landing on items later contested — the hesitation mechanism works. Raw agreement against the curation-derived key: 18/32 (56%).
- **Adjudication (all 14 disagreements walked with full texts):** the raw 56% decomposed cleanly. (a) **Nine of Ian's tags stood** — the *key* was wrong: curation categories are mining provenance assigned before the protocol existed, and several fail the protocol's own decision rules ([209] is the protocol's verbatim example of what NOT to tag `negation`, yet was keyed `negation`). (b) **Five tags revised** on discussion: [840]→`other`, [10617]→`other`, [15969]→`other`, [7919]→`negation`, [10866]→`sarcasm`. Blind first-pass vs adjudicated finals: 27/32 (84%). Both columns preserved in the pilot sheet (`tag` = blind, `tag_final` = adjudicated).
- **Protocol amended to v1.1** — three rules sharpened, each traceable to specific pilot items: (1) **`sarcasm` requires polarity inversion** — surface-positive vocabulary carrying a negative verdict (or vice versa), the property that actually fools a bag-of-words model; insult or exasperation in surface-negative vocabulary is plain sentiment. (2) **`negation` carriage is judged on the whole review** — a buried reversal clause doesn't qualify; hope-then-disappointment arcs do. (3) **`mixed` requires the verdict to rest on the weighing** — stacked weak concessions around an unwavering verdict don't count. Anchors updated: [17596] moved from `mixed` to `negation` anchor; [24446] retired as an anchor; [6035] added for sarcasm.
- **Noise category kept, expected count ~zero (pre-registered):** the pilot retired the only specimen — [21360]'s curator, tagging blind, found an arguable verdict in it. Genuine `noise` now means defective text (wrong language, pure markup, opinion-free synopsis). A zero in the real run is a corpus finding, not a protocol failure.
- **Role separation made explicit:** `golden_set.csv: category` is curation/stratification provenance (frozen — it built the judge's stratified readout) and is *not* tag ground truth; the protocol tags are the taxonomy instrument. The two disagree by design in places, which the pilot surfaced rather than hid.
- Final adjudicated distribution on the golden 32: `other` 19, `sarcasm` 6, `negation` 4, `mixed` 3, `noise` 0.

## 2026-07-23

- **Serving switched to llama-swap routing:** the judge endpoint is now the llama-swap router (`127.0.0.1:9090`, model selected by the request's `model` field) instead of a hand-launched llama-server on `:8080`. Same llama.cpp backend, launcher, and flags underneath — only the `.env` base URL changed. First request after idle triggers a cold model load (~1–2 min); subsequent calls are unaffected (32 reasoning-off reviews in 13 s once warm).
- **Prompt v2 candidate evaluated (freeze test).** A rating-inference framing was proposed post-freeze: tell the judge the labels derive from the reviewer's own 1–10 star rating (≥7 positive, ≤4 negative), instruct it to infer the rating; judge the reviewer's verdict rather than the writing or the film; explicit guidance that mixed reviews resolve by weighing and irony resolves by intended meaning. Evaluated head-to-head against frozen v1 on the golden 32 under the frozen config (temp 0, reasoning off, schema-constrained, blind): **identical predictions on all 32 items** — same 31/32 (96.9%) overall, 21/22 (95.5%) hard, same single miss. Notably, [17596] contains an explicit in-text grade ("I give this a B-"), the one signal the rating-inference framing was best positioned to exploit — the judge still read the hedged prose as negative. **Decision: freeze upheld; prompt v1 stands.** At temp 0 with forced-binary output, the extra task framing moved no argmax; the miss remains the irreducible label-noise exemplar, not a prompt deficiency.
- **Prompt v3 merged and adopted as the FINAL frozen prompt.** Since v1 and v2 scored identically, the final wording combines the best of each on prompt-quality grounds rather than measured difference: v1's adjudicator role, forced-choice-even-for-mixed clause, and explicit JSON instruction; v2's rating-inference grounding, judge-the-verdict-not-the-writing rule, and the mixed/irony guidance. Verbatim system prompt:

  > *"You are a sentiment adjudicator for IMDB movie reviews. Each review's author also gave the film a star rating from 1 to 10, and the dataset's label is derived from that rating (>= 7 positive, <= 4 negative). Read the review, infer the reviewer's verdict, and classify it as positive or negative.*
  >
  > *Judge the reviewer's verdict on the film, not the quality of their writing and not your own opinion of the film. Where a review is mixed, decide which way the weighing lands. Where praise or criticism is ironic, the intended meaning governs, not the surface words. You must choose exactly one label, even for mixed reviews. Respond in JSON."*

  Regression-checked on the golden 32 under the frozen config (temp 0, reasoning off, schema-constrained, blind): **31/32 (96.9%) overall, 21/22 (95.5%) hard, predictions identical to v1 on every item** (same miss, [17596]). The JSON instruction is belt-and-braces — output is still schema-enforced server-side (`response_format` forced-binary enum), so the prompt line documents intent rather than doing the enforcement. **Prompt v3 supersedes v1 as the frozen judge prompt for notebook 05; all other frozen-config decisions (reasoning off, temp 0, blind, schema) unchanged.**
- **Notebook 05 scaffolded — the frozen config moves into its contractual home.** Until now the judge request code existed only as scratch scripts (lost once already to a temp-directory wipe) with the dev log as the durable spec. `notebooks/05_divergence_judge.ipynb` now carries: the frozen configuration (prompt v3 verbatim, per-request `temperature=0` and `enable_thinking=false`, forced-binary schema, provider-agnostic `.env` endpoint), a checkpointed `adjudicate()` runner (one CSV row per verdict, interrupt-safe — re-running skips completed ids), and a live golden-set evaluation. Executed end-to-end: fresh run reproduced the frozen result (31/32, 21/22 hard, same miss) and committed it as `outputs/tables/05-judge_golden_eval.csv`; immediate re-run confirmed resume semantics ("32 checkpointed, 0 to run" — zero API calls). Disagreement sections are guarded — they self-report what they await (02/03 prediction files) so the scaffold runs clean top to bottom. Output prefix `05-judge_*` adopted, matching 01's pipeline-ordering convention.
